# 面試展示講稿

## 一句話介紹

這是一個遊戲商城 demo，使用 Next.js 做前端、Express 做 API，包含會員登入、商品瀏覽、購物車、訂單付款、管理後台，以及可回答商城問題的 AI 客服。

## 30 秒版本

這個專案模擬一個小型遊戲商城。使用者可以瀏覽商品、加入購物車、建立訂單，接著用 Stripe 測試卡或 Demo 快速付款完成流程。後台管理員可以查看商品與訂單。AI 客服則用簡單 RAG，把商品資料與商城規則整理給使用者，例如付款、退款、配送與商品推薦。整體重點不是做成正式電商，而是展示我能把前端流程、後端 API、付款狀態與 AI 問答串成一個可操作的完整 demo。

## Demo Flow

1. 從首頁選一個商品，進商品詳情。
2. 加入購物車，確認品項與數量。
3. Checkout 建立訂單。
4. 到訂單中心付款。
5. 使用 Stripe 測試卡或 Demo 快速付款。
6. 回到訂單中心確認狀態變成付款完成。
7. 用 admin 登入後台，查看訂單與商品。
8. 打開 AI 客服，詢問「怎麼付款」、「可以退款嗎」、「推薦便宜的遊戲」。

## 前後端分工

前端用 Next.js / React / TypeScript 實作頁面狀態、購物車操作、訂單 UI、付款入口與 AI 客服介面。後端用 Express 處理會員驗證、商品資料、購物車、訂單狀態、付款模擬與 AI 客服資料檢索。資料目前存在 SQLite，方便本機 demo 與測試。

## 付款流程

Checkout 會先建立一筆 pending 訂單。使用者可以走 Stripe test mode，使用測試卡號完成付款；如果沒有設定 Stripe key，也可以使用 Demo 快速付款讓展示流程不中斷。付款成功後訂單狀態會變成 paid，付款失敗可以重新付款，退款則會把狀態改成 refunded。

## AI / RAG 設計

AI 客服定位是商城客服，不是通用聊天機器人。它會先從商品 catalog 與靜態商城知識中找資料，例如付款、退款、配送、帳號、願望清單。若有命中資料，畫面會標示「根據商城資料回答」並顯示來源。正式環境優先使用 OpenAI；本機可以用 Ollama 作為 fallback；兩者都沒有時，仍會回固定客服回覆，確保 demo 可展示。

## Ollama 與正式部署差異

Ollama 適合本機學習與 demo，模型跑在自己的電腦上。部署到 Vercel 或 Render 時，別人不會自動連到你電腦上的 Ollama。正式環境如果要穩定提供 AI 客服，建議設定 OpenAI key，或另外部署一個可被正式環境存取的模型服務。

## 目前限制

SQLite 只適合 demo persistence，正式環境會改成 PostgreSQL 或 managed database。圖片上傳目前是 local disk，正式環境會改成 Vercel Blob、S3 或 Cloudinary。Demo 快速付款只代表展示流程完成，不代表真實金流。AI 客服目前只做客服導引與商品問答，不直接操作訂單、付款或退款。

## 正式化方向

第一步會把資料庫換成正式 managed database，補完整 migration 與資料備份。第二步會把付款 webhook 做完整驗證，讓訂單狀態完全以 Stripe 事件為準。第三步會把 AI 客服升級成更完整的 RAG，例如向量搜尋、訂單查詢工具與更精準的商品推薦，但仍會保留權限與操作邊界，避免 AI 直接執行敏感動作。
