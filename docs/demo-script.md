# Demo 展示腳本

這份文件是展示時可照順序操作的腳本，口吻偏現場說明，不是完整技術文件。

## 開場

這是一個 Steam 風格的商城 demo。主流程包含商品瀏覽、購物車、checkout、訂單、付款、管理後台，以及 AI 商城客服。

我這次整理的重點不是新增大量功能，而是讓整個流程可以穩定展示，並把 AI 做成可解釋的客服、商品推薦與購買決策功能。正式環境不放 OpenAI key 時，商品頁、比較頁與購物車也有 Browser-side AI / 商品資料 fallback，可以展示零 API 成本的 AI 互動入口。

## 展示前準備

```bash
npm run dev
```

確認：

- Frontend：http://localhost:3000
- API：http://localhost:4000

若 4000 被佔用：

```powershell
netstat -ano | Select-String ':4000'
Stop-Process -Id <PID> -Force
```

## 5 分鐘 AI Demo Checklist

這段是面試時最推薦照著跑的快速版，目標是讓對方在 5 分鐘內看到 AI 如何接進真實商城流程。

1. **首頁 AI 入口**
   - 使用 Demo 快速登入。
   - 從首頁指出「AI 助理」三個入口：商品決策、商品比較、AI 客服。
   - 可說：AI 沒有被做成孤立 dashboard，而是放在使用者會自然走到的位置。

2. **商品頁：幫我判斷**
   - 進任一商品詳情。
   - 點「幫我判斷」。
   - 看 AI 來源標籤：`本機 AI 整理` 或 `商品資料整理`。
   - 可說：沒有 OpenAI key 時仍可展示，因為會退回商品資料與偏好分析。

3. **比較頁：幫我選一款**
   - 加入 2 到 3 款商品比較。
   - 到比較頁點「幫我選一款」。
   - 展示它會收斂成一個建議商品，而不是只列差異。

4. **購物車：結帳前檢查**
   - 加入商品到購物車。
   - 點「檢查」。
   - 展示 AI 會看總價、版本、數量與偏好，提醒送出前是否需要調整。
   - 可說：它只做建議，不會替使用者刪商品或建立訂單。

5. **訂單流程**
   - Checkout 建立 `pending` 訂單。
   - 使用 Stripe 測試卡或 Demo 快速付款。
   - 回訂單中心確認付款完成。

6. **AI 客服 / RAG**
   - 到 AI 商城客服依序問：
     - `預算 30 美金，想玩 RPG，哪一款適合？`
     - `這筆訂單接下來怎麼辦？`
     - `可以退款嗎？`
   - 打開「顯示 RAG 依據」看 sources 與 score。
   - 可說：客服回答不是黑盒聊天，會先從商品、政策或訂單資料找依據。

## 展示流程

### 1. Demo 快速登入

先使用 Demo 快速登入，避免面試時花時間註冊帳號。

可說：

> 這裡提供 demo user，方便快速進入完整會員流程。正式環境會使用正常註冊與登入。

### 2. 商品與購物車

首頁有一個簡潔的「AI 助理」入口，可快速帶到商品決策、商品比較與 AI 客服。

可說：

> 我沒有做一個很重的 AI dashboard，而是把 AI 放在使用者真的會用到的三個入口：選商品、比較商品、問客服。

接著從首頁選一款商品，進商品詳情，確認版本與庫存後加入購物車。

可說：

> 商品有版本與庫存概念，checkout 時會檢查庫存，避免超賣。

在商品頁點「幫我判斷」。

可說：

> 這裡是零 API 成本的商品決策助理。瀏覽器支援內建 AI 時，會在使用者裝置上產生購買建議；不支援時，會用商品資料與近期偏好產生穩定建議，所以 demo 不依賴 OpenAI 額度。

畫面會顯示目前分析方式：

- `本機 AI 可用`
- `可啟用本機 AI`
- `目前使用商品資料分析`

AI 回覆區也會用一致的標籤顯示來源：

- `AI 整理`
- `本機 AI 整理`
- `商品資料整理`

也可以加入 2 到 3 款商品比較，進比較頁點「幫我選一款」。

可說：

> 這裡不是只列商品差異，而是把比較結果收斂成一個建議選擇。沒有本機 AI 時，也會根據商品資料、價格、庫存與近期偏好給出可展示的決策結果。

### 3. Checkout 建立訂單

進購物車後，先點「檢查」展示 AI 購物車檢查，再 checkout 建立一筆 `pending` 訂單。

可說：

> 這裡是結帳前檢查。它會看總價、重複版本、數量和近期偏好，提醒使用者送出前要不要調整。它不會替使用者刪商品，只提供建議。

> Checkout 先建立訂單，不直接視為付款完成。付款狀態會另外處理，這比較接近真實電商流程。

### 4. 付款

到訂單中心或訂單詳情付款。

如果有 Stripe key：

- 使用測試卡 `4242 4242 4242 4242`
- 任意未來日期
- 任意 CVC

如果沒有 Stripe key：

- 使用 Demo 快速付款

可說：

> Demo 快速付款是展示 fallback。正式化時會以 Stripe webhook 作為付款狀態來源。

### 5. 訂單與管理後台

回訂單中心確認訂單已付款，再進 admin 後台查看訂單與營收。

可說：

> 使用者端看到自己的訂單，管理端可以看到整體訂單與營收摘要。權限用 JWT role 控制。

### 6. AI 商城客服

進 AI 商城客服，依序問：

```text
預算 30 美金，想玩 RPG，哪一款適合？
Elden Ring 跟 The Witcher 3 哪個適合？
幫我檢查購物車適不適合結帳
這筆訂單接下來怎麼辦？
可以退款嗎？
```

可說：

> AI 客服不是直接把問題丟給模型。它會先從商品、政策或訂單資料找依據，再產生回答。

展示重點：

- 商品決策會顯示商品卡，使用者可以直接點「查看商品」。
- 商品比較會顯示比較表，讓 AI 回答不是只有一段文字。
- 購物車健檢會讀登入者自己的購物車，整理總價與結帳前建議。
- 訂單售後會讀登入者最近一筆訂單，依狀態提示付款、出貨或退款下一步。
- 一般退款問題仍然走客服知識，不需要登入。

### 7. RAG Debug

本機可以打開「顯示 RAG 依據」，再問：

```text
可以退款嗎？
推薦便宜的遊戲
```

畫面會顯示：

- retriever：`local-hybrid`
- 命中的文件
- score
- exact / title / content / tags / intent breakdown

可說：

> 這個 debug 面板是為了展示與調整 RAG。它能看出 AI 回答根據哪些資料，不是黑盒聊天。

## AI 架構講法

目前是 local hybrid RAG，加上一個 Browser-side AI POC：

```text
使用者問題
  -> 建立 query
  -> local hybrid retriever 找 FAQ / policy / catalog
  -> score by exact / title / content / tags / intent
  -> 組成 context
  -> OpenAI / Ollama 生成自然回覆
  -> fallback 確保 demo 可用
```

補充：

> 因為 demo 資料量不大，所以先不用 vector database。架構上有保留 Retriever 介面，之後可以把 local retriever 換成 embeddings + pgvector 或 Pinecone。

> 另外，商品頁、比較頁與購物車的本機決策助理是為了處理展示成本問題。OpenAI 品質好但會花錢，Ollama 本機可用但部署後不會自動存在；所以我加了一層瀏覽器端 AI，能用就用，不能用就回到商品資料分析，而且畫面會清楚顯示目前使用哪一種分析方式。

## 個人化推薦講法

個人化推薦會參考：

- 願望清單
- 購物車
- 曾經買過的商品
- 商品價格與庫存

可說：

> 個人化排序在後端完成，個人訂單摘要不送給外部模型。AI 只負責把候選商品和原因整理成比較自然的客服回覆。

## AI 功能展示講法

可以用這段串起來：

> 我把 AI 放在使用者真的會遇到的商城流程裡，而不是做一個孤立聊天室。選商品時，它可以做預算推薦、商品比較與購買決策；結帳前，它可以檢查購物車；付款後，它可以依訂單狀態說明下一步。這些功能都先整理資料和權限，再讓 AI 或 fallback 產生自然回覆，所以不會讓模型直接操作付款、退款或訂單。

## 專案限制與下一步

可以主動說：

> 目前這是展示版，所以 SQLite、local upload、Demo 快速付款都不是正式上線方案。正式化會改 managed database、外部圖片儲存、Stripe webhook。AI 方向會先保留 Browser-side AI 作為零成本入口，再依需求接 hosted model、embeddings 與 vector database。

## 收尾

可用這段收尾：

> 這個專案的重點是把一般商城流程和 AI 客服整合起來。AI 不是獨立聊天功能，而是圍繞商品、訂單、付款與退款流程，並且可以透過 RAG debug 看到回答依據，方便後續擴充與維護。
